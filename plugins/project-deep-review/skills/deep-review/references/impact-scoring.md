# Impact Scoring Guide

## 등급 정의

### 🔴 MUST (필수 개선)

**기준**: 안 하면 실질적 손해가 발생하는 문제

**판단 질문**:
- 이 문제로 사용자가 서비스를 사용할 수 없는가?
- 보안 사고가 발생할 수 있는가?
- 매출/수익에 직접적 영향을 주는가?
- Core Web Vitals이 "Poor" 범위인가?

**예시**:
| 영역 | MUST 예시 |
|------|----------|
| 기술스택 | 보안 취약점이 알려진 패키지, 지원 종료 버전 |
| UX | 깨진 사용자 플로우, WCAG 위반 (법적 리스크) |
| 성능 | LCP > 4s, 메모리 누수, 번들 > 500KB gzip |
| 해자 | 결제 오류, 데이터 유실 가능성, 인증 취약점 |
| 코드 | 타입 안전성 결여로 인한 런타임 에러 리스크 |

### 🟠 SHOULD (권장 개선)

**기준**: 하면 체감 가능한 개선이 되는 문제

**판단 질문**:
- 측정 가능한 지표가 개선되는가? (성능, 전환율, 이탈률)
- 개발팀의 생산성에 유의미한 영향을 주는가?
- 사용자가 불편함을 느끼지만 서비스 사용은 가능한가?

**예시**:
| 영역 | SHOULD 예시 |
|------|------------|
| 기술스택 | 다음 메이저 버전으로 마이그레이션 (파괴적 변경 없이) |
| UX | 혼란스러운 네비게이션, 부족한 에러 메시지 |
| 성능 | LCP 2.5-4s, 번들 200-500KB, 불필요한 리렌더링 |
| 해자 | 경쟁사 대비 부족한 핵심 기능, 느린 온보딩 |
| 코드 | 핵심 비즈니스 로직 테스트 부재, 높은 코드 중복 |

### ⚪ COULD (이론적 개선) — 보고하지 않음

**기준**: 이론적으로는 더 나아질 수 있지만 실질적 차이가 미미한 개선

**예시**: 미세한 성능 개선(10ms), 코드 스타일 변경, 최신 트렌드 따라가기

### 💪 STRENGTH (강점)

**기준**: 현재 잘 되고 있어서 유지해야 하는 것

**중요**: 모든 영역에서 최소 3개 이상 강점을 반드시 포함해야 합니다.
강점을 인식하는 것은 편향 방지의 핵심입니다.

## 등급 판정 시 주의사항

1. **규모 고려**: 1인 사이드 프로젝트와 팀 프로덕션의 기준이 다름
2. **단계 고려**: MVP 단계와 성숙 서비스의 기준이 다름
3. **도메인 고려**: B2C와 B2B, 이커머스와 SaaS의 우선순위가 다름
4. **비용 고려**: 개선에 필요한 시간/리소스 대비 효과를 항상 언급
