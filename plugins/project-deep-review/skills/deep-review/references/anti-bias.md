# Anti-Bias Framework

## "개선 편향"이란?

리뷰를 요청받으면 "문제를 찾아야 한다"는 압박으로 불필요한 개선까지 제안하는 현상.

## 편향 방지 체크리스트

### Before Analysis (분석 전)

- [ ] 프로젝트의 **현재 단계**를 파악했는가? (MVP, Growth, Mature)
- [ ] 프로젝트의 **규모와 팀 구성**을 파악했는가?
- [ ] 프로젝트의 **도메인**을 이해했는가?

### During Analysis (분석 중)

- [ ] 각 발견 사항에 대해 "이것을 안 하면 실제로 무슨 일이 일어나는가?" 질문했는가?
- [ ] 강점(STRENGTH)을 적극적으로 찾고 있는가?
- [ ] 구현 비용 대비 효과를 고려하고 있는가?

### After Analysis (분석 후)

- [ ] STRENGTH가 영역당 최소 3개 포함되었는가?
- [ ] COULD 등급이 결과에서 제거되었는가?
- [ ] 모든 MUST/SHOULD에 근거(URL, 데이터)가 있는가?
- [ ] 프로젝트 규모/단계에 과도한 제안이 포함되지 않았는가?

## 흔한 편향 패턴

### 1. 트렌드 편향
"최신 기술/패턴이 아니므로 바꿔야 한다"
→ **현재 기술이 잘 작동하면 바꿀 이유가 없습니다**

### 2. 완벽주의 편향
"이론적으로 더 나은 구조가 있으므로 리팩토링해야 한다"
→ **현재 구조가 문제를 일으키지 않으면 그대로 두세요**

### 3. 스케일 편향
"대규모 시스템에서는 이렇게 해야 한다"
→ **현재 프로젝트 규모에 맞는 제안을 하세요**

### 4. 개인 취향 편향
"나라면 이렇게 했을 것이다"
→ **객관적 근거 없는 주관적 선호는 제안하지 마세요**

### 5. 포괄성 편향
"모든 영역에서 개선점을 찾아야 한다"
→ **특정 영역에 개선점이 없으면 없다고 보고하세요**

## 올바른 리뷰 태도

```
❌ "이것을 개선할 수 있다"
✅ "이것을 개선해야 하며, 그 이유는 {근거}이고, 안 하면 {실질적 손해}이다"

❌ "최신 버전을 사용해야 한다"
✅ "최신 버전에 {이 프로젝트에 필요한 구체적 기능/보안 패치}가 있다"

❌ "테스트가 부족하다"
✅ "{이 비즈니스 로직}에 테스트가 없어서 {이런 종류의 버그}가 발생할 수 있다"

❌ "디자인이 좋아질 수 있다"
✅ "{이 사용자 플로우}에서 {구체적 문제}로 인해 {사용자 이탈/혼란}이 발생할 수 있다"
```
